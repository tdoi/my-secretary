#### 所感
境界を意識すること。                                                        
                                                                            
LLM を使った開発では、大部分の実装においては、実装速度そのものは制約ではなくなる。
しかし、システムの品質という観点で考えれば、まだ人が注意を払うべきポイントがある。それが、境界の設計である。
データモデリングを間違えば、SQLが複雑になる、非効率になり、見えにくい負債をかかえこむ。
API 設計を間違えば、後々の修正コストを増やしたり、無駄に複雑な API 呼び出しによるパフォーマンスの低下を引き起こす。
実装は任せられるが、境界は完全に任せるのはリスクがある。
実際、LLM の計画に対して、修正の指示を出すのはこの点がほとんどだと思う。

システムとしての境界も同様である。
各システムの責任の境界を設計することは、重要なUXの意思決定である。
機能を実現するための論理性は担保できたとしても、設計思想そのものは、まだ完全に任せにくいポイントである。
元々の思想として、SaaS is Dead と言われ始めた要因の1つは、CRUD + ちょっとした機能は、容易に代替できるものになってしまったからである。
どんな機能を提供するのかで設計されたUXは、まさに無価値である。
システム群全体として、どんな価値を提供するのかを意識しなければならない。

各システムの責務は意識して作ってきているつもりでいるが、価値の提供の仕方としての UX は再度、検討する必要がある。
